# JUNO Audit Report: WO-004 Remotes, Pull Requests, and PR Detail Modal

**Auditor:** JUNO (Quality Auditor)
**Date:** 2026-02-03
**Scope:** Remotes button, Pull Requests button, PullRequestDetailModal, supporting IPC channels and services
**Rating:** GOOD (82/100)

---

## Executive Summary

The WO-004 implementation delivers functional Remotes and Pull Requests dropdowns in the DevelopmentScreen toolbar, plus a PullRequestDetailModal with parallel data fetching, unified timeline, and comment submission. The IPC layer is fully typed and all 5 new handlers are registered. Code quality is generally solid with consistent patterns, but there are several edge cases, one dead code issue, and a few missing error-handling paths that need test coverage.

## 1. Findings by File

### 1.1 DevelopmentScreen.tsx

#### F-DS-001: `extractOwnerRepo` regex edge cases (Severity: MEDIUM)
**Location:** Line 41
**Pattern:** `/github\.com[/:]([\w.-]+)\/([\w.-]+?)(?:\.git)?$/`
**Issue:** The regex handles HTTPS URLs and SSH `git@github.com:owner/repo` patterns, but:
- Returns `null` for non-GitHub URLs (GitLab, Bitbucket) -- expected but untested
- The lazy `[\w.-]+?` for repo name could mismatch on edge-case repo names with trailing dots
- Does NOT handle URLs with trailing slashes (`https://github.com/owner/repo/`)
- Does NOT handle URLs with query strings or fragments

**Test cases needed:** Valid HTTPS, valid SSH, `.git` suffix, trailing slash, non-GitHub URL, empty string, `null`/`undefined` input protection

#### F-DS-002: Remotes dropdown re-fetches on every open (Severity: LOW)
**Location:** Lines 63-78
**Issue:** The `useEffect` fires whenever `activeDropdown === 'remotes'`, meaning every toggle-open re-fetches remotes from disk. There is no caching or staleness check. Not a bug, but an inefficiency worth noting. No `remotesError` state exists -- errors silently set remotes to `[]`, which displays "No remotes configured" (misleading when the error is actually a git failure).
**Test cases needed:** Fetch success, fetch failure (verify empty array fallback), `localPath` is undefined/null

#### F-DS-003: Pull Requests dropdown silent failure paths (Severity: MEDIUM)
**Location:** Lines 81-107
**Issue:** When `targetUrl` is falsy (line 85) or `extractOwnerRepo` returns null (line 88-89), the effect returns early without setting any loading/error state. If the dropdown was previously showing data from another contribution, stale data remains visible.
**Test cases needed:** No `upstreamUrl` and no `repositoryUrl`, invalid URL that fails parsing, successful fetch, fetch error, stale data after switching contributions

#### F-DS-004: Button color logic is deeply nested and complex (Severity: LOW)
**Location:** Lines 243-262
**Issue:** The ternary chain for button styling is 6 levels deep. While functionally correct, it is difficult to verify visually and prone to regression. The Remotes button highlights when `contribution.isFork && contribution.remotesValid`. The PR button highlights green for `prStatus === 'open'`, primary for `prStatus === 'merged'`, and default for `prStatus === 'closed'` or no status.
**Test cases needed:** All 8 color state combinations (remotes: fork+valid, fork+invalid, not-fork; PRs: open, merged, closed, no-status, active dropdown)

#### F-DS-005: Modal render uses IIFE pattern (Severity: LOW)
**Location:** Lines 378-389
**Issue:** The `{selectedPR && (() => { ... })()}` pattern works but is unusual. If `extractOwnerRepo` returns null, the modal silently doesn't render with no user feedback. The user clicks a PR, dropdown closes, but nothing happens.
**Test cases needed:** Selected PR with valid URL, selected PR with invalid URL (no modal), closing modal sets selectedPR to null

#### F-DS-006: `isMounted` ref set in multiple effects (Severity: LOW)
**Location:** Lines 58, 154-168
**Issue:** `isMounted.current = true` is set at line 58 (initialization) AND again at line 155 (inside the auto-start effect). The cleanup at line 163 sets it to false. This pattern is correct for strict mode double-mount protection, but the `isMounted` ref is also used by the remotes and PR fetch effects (lines 68-75, 95-105), which do NOT set `isMounted.current = true` in their own effects. This is fine because those effects run after the initial mount, but it is worth testing the unmount scenario.

### 1.2 PullRequestDetailModal.tsx

#### F-PR-001: `reviewCommentsByFile` is dead code (Severity: LOW)
**Location:** Lines 200-204
**Issue:** The `reviewCommentsByFile` grouping is computed on every render but NEVER used in the JSX. This is dead code that should be removed or the grouped-by-file view should be implemented.
**Test cases needed:** Verify this code path does not affect rendering (it doesn't, but tests should confirm no side effects)

#### F-PR-002: `fetchData` is not memoized and is a dependency concern (Severity: LOW)
**Location:** Lines 126-151, 153-159
**Issue:** `fetchData` is defined as a plain async function inside the component. The `useEffect` at line 153 does NOT list `fetchData` as a dependency (it depends on `owner` and `repo` via closure). This works because `owner` and `repo` ARE listed as dependencies, but it triggers the React exhaustive-deps lint warning. The retry button at line 233 calls `fetchData(pr.number)` directly, which captures stale `owner`/`repo` if they change between the initial render and the retry click (extremely unlikely in practice).
**Test cases needed:** Initial data fetch, retry after error, component unmount during fetch (isMounted cleanup)

#### F-PR-003: Date objects from IPC are serialized as strings (Severity: HIGH)
**Location:** Lines 184, 191, 194, 197
**Issue:** The `PRComment.createdAt`, `PRReview.submittedAt`, and `PRReviewComment.createdAt` fields are typed as `Date` in the interfaces, but IPC serialization (Electron's structured clone) will pass them as `Date` objects from the main process. However, if the data passes through JSON serialization at any point (e.g., preload bridge), dates become strings. The code wraps them in `new Date()` at lines 184, 191, 194, which is defensive and correct. But `formatDate` at line 106 ALSO wraps in `new Date(date)`, creating a double-wrap. This is harmless but indicates uncertainty about the actual type at runtime.
**Test cases needed:** Timeline items with Date objects, timeline items with ISO string dates, `formatDate` with both types, sorting with mixed types

#### F-PR-004: Timeline sorting assumes valid dates (Severity: MEDIUM)
**Location:** Line 197
**Issue:** `timeline.sort((a, b) => a.date.getTime() - b.date.getTime())` will produce `NaN` if any date is invalid (`new Date('invalid').getTime()` returns `NaN`). This would corrupt sort order silently.
**Test cases needed:** Timeline with valid dates, timeline with null/undefined/invalid date strings from API

#### F-PR-005: Comment submission has no error feedback to user (Severity: MEDIUM)
**Location:** Lines 161-176
**Issue:** `handleSubmitComment` catches errors and logs to console but does NOT set any error state or show a toast/notification to the user. The comment textarea is cleared on success, but on failure the user gets no indication that the comment was not posted. Additionally, if the refresh of comments (line 169) fails, that error is also swallowed by the outer catch.
**Test cases needed:** Successful comment submission, failed submission (verify textarea retains content), failed comment refresh after successful submission

#### F-PR-006: `onOpenChange` handler receives boolean, not void (Severity: LOW)
**Location:** Line 207
**Issue:** `<Dialog open={!!pr} onOpenChange={onClose}>` -- Radix Dialog's `onOpenChange` passes a boolean (the new open state). `onClose` expects no arguments. This works because calling `onClose(false)` still executes the function, but it is semantically incorrect. If `onClose` ever checks its arguments, this could break.
**Test cases needed:** Close via overlay click, close via escape key, close via explicit button

#### F-PR-007: Reviews with empty body and non-COMMENTED state still render (Severity: LOW)
**Location:** Lines 188-189, 306-310
**Issue:** The filter at line 189 skips reviews with `!r.body && r.state === 'COMMENTED'`. But reviews with `!r.body` and states like `APPROVED` or `CHANGES_REQUESTED` still render -- they show the badge but no body content, which is the correct GitHub behavior. The `{r.body && ...}` guard at line 306 handles this gracefully.
**Test cases needed:** APPROVED review with no body, CHANGES_REQUESTED with body, COMMENTED with body, COMMENTED with no body (filtered), PENDING review, DISMISSED review

#### F-PR-008: No pagination for comments/reviews (Severity: LOW)
**Location:** Lines 131-136
**Issue:** All four IPC calls use `per_page: 100`. PRs with >100 comments, reviews, or review comments will be truncated silently. No pagination or "load more" UI exists.
**Test cases needed:** Not directly testable in unit tests, but the service methods should be tested with pagination awareness

### 1.3 github-rest.service.ts

#### F-GH-001: All PR methods return `Promise<any[]>` (Severity: MEDIUM)
**Location:** Lines 430-520
**Issue:** `listPRComments`, `listPRReviews`, and `listPRReviewComments` all return `Promise<any[]>`. The channel types in `channels.ts` ARE properly typed, but the service layer loses type safety. This means the mapping logic (lines 444-451, 475-482, 506-514) is not validated at compile time.
**Test cases needed:** Each method should be tested to verify the returned shape matches the channel type definitions

#### F-GH-002: `listPRReviews` falls back to `new Date()` for missing `submitted_at` (Severity: MEDIUM)
**Location:** Line 481
**Issue:** `submittedAt: review.submitted_at ? new Date(review.submitted_at) : new Date()` -- If `submitted_at` is null (which happens for PENDING reviews), the fallback to `new Date()` creates a timestamp that is the current time, not the review creation time. This would sort PENDING reviews to the end of the timeline (which is arguably correct but not semantically accurate).
**Test cases needed:** Review with valid `submitted_at`, review with null `submitted_at` (PENDING state)

#### F-GH-003: `listPRReviewComments` line fallback logic (Severity: LOW)
**Location:** Line 512
**Issue:** `line: comment.line || comment.original_line || null` -- Uses `||` which treats `0` as falsy. If a review comment is on line 0 (unlikely but possible in some diff contexts), it would fall through to `original_line` or `null`. Should use `??` instead of `||`.
**Test cases needed:** Comment on line 1, comment on line 0, comment with null line and valid original_line, comment with null for both

#### F-GH-004: `per_page: 100` hardcoded with no pagination (Severity: LOW)
**Location:** Lines 442, 470, 500
**Issue:** All three methods hardcode `per_page: 100`. For PRs with extensive discussion (>100 items), data is silently truncated.
**Test cases needed:** Not a unit test concern, but should be documented as a known limitation

#### F-GH-005: `create-pr-comment` handler uses `createIssueComment` (Severity: INFO)
**Location:** `index.ts` line 367
**Issue:** The `github:create-pr-comment` IPC handler calls `gitHubRestService.createIssueComment()`. This is CORRECT because GitHub's API treats PR comments as issue comments. But the naming mismatch could confuse developers. Worth noting, not a bug.
